This guide will walk you through your first steps with `ee`, from initializing a project to managing environment variables across different environments.

## Step 1: Initialize Your Project

Navigate to your project directory and initialize ee:

```bash
cd my-web-app
ee init
```

This creates a `.ee` configuration file in your project root with:
- Default project name (from directory name)
- Standard environments (development, production)
- Default inline schema with common variables

**What gets created:**

`.ee` file:
```json
{
  "project": "my-web-app",
  "schema": {
    "variables": {
      "NODE_ENV": {
        "name": "NODE_ENV",
        "type": "string",
        "title": "Node environment",
        "required": false,
        "default": "development"
      },
      "PORT": {
        "name": "PORT",
        "type": "number",
        "title": "Server port",
        "required": false,
        "default": "3000"
      },
      "DEBUG": {
        "name": "DEBUG",
        "type": "boolean",
        "title": "Debug mode",
        "required": false,
        "default": "false"
      }
    }
  },
  "environments": {
    "development": {
      "sheets": [".env.development"]
    },
    "production": {
      "sheets": [".env.production"]
    }
  }
}
```

Sample `.env` files are also created:
- `.env.development`
- `.env.production`

## Step 2: Configure Your Environments

Edit the generated `.env` files with your actual configuration:

**`.env.development`**:
```bash
# .env.development environment configuration
# Edit the values below according to your needs

# title: Node environment
# type: string
# default: development
NODE_ENV=development

# title: Server port
# type: number
# default: 3000
PORT=3000

# title: Debug mode
# type: boolean
# default: false
DEBUG=true

# Add your custom variables
DATABASE_URL=postgres://localhost:5432/myapp_dev
API_KEY=dev-api-key-123
```

**`.env.production`**:
```bash
# .env.production environment configuration

NODE_ENV=production
PORT=8080
DEBUG=false

DATABASE_URL=postgres://prod-db:5432/myapp
API_KEY=prod-api-key-xyz
```

## Step 3: Apply Your Environment

### Preview what will be applied

```bash
# Dry-run to see what variables will be loaded
ee apply development --dry-run
```

Output:
```
Applying environment 'development'
Environment variables that would be applied:
export NODE_ENV=development
export PORT=3000
export DEBUG=true
export DATABASE_URL=postgres://localhost:5432/myapp_dev
export API_KEY=dev-api-key-123
```

### Load into a new shell

```bash
# Start a new shell with development environment loaded
ee apply development
```

You'll see:
```
Applying environment 'development'
Starting shell with environment variables applied
Shell: /bin/bash
Applied 5 environment variables
```

Now check your variables:
```bash
echo $NODE_ENV    # development
echo $PORT        # 3000
echo $DEBUG       # true
```

Exit the shell to return to your original environment:
```bash
exit
```

### Run a command with the environment

```bash
# Run your application with development config
ee apply development -- npm start

# Run with production config
ee apply production -- npm start

# Run tests
ee apply development -- npm test
```

## Step 4: Advanced Configuration

### Create a Custom Schema

Create a schema for more complex validation:

```bash
ee schema create web-service \
  --description "Web service configuration" \
  --variable "PORT:number:Server port:true:3000" \
  --variable "DATABASE_URL:url:Database connection:true" \
  --variable "DEBUG:boolean:Debug mode:false:false" \
  --variable "API_KEY:string:API authentication key:true"
```

### Update Your Project to Use the Schema

Edit your `.ee` file to reference the schema:

```json
{
  "project": "my-web-app",
  "schema": {
    "ref": "local://web-service"
  },
  "environments": {
    "development": {
      "sheets": [".env.development"]
    },
    "production": {
      "sheets": [".env.production"]
    }
  }
}
```

### Use Config Sheets Instead of .env Files

Create a config sheet:

```bash
ee sheet create dev-config \
  --schema web-service \
  --value PORT=3000 \
  --value DATABASE_URL=postgres://localhost:5432/myapp_dev \
  --value DEBUG=true \
  --value API_KEY=dev-key-123
```

Update `.ee` to use the config sheet:

```json
{
  "environments": {
    "development": {
      "sheets": ["local://dev-config"]
    }
  }
}
```

### Stack Multiple Sheets

Create a base configuration and environment-specific overrides:

**Base .env file** (`.env`):
```bash
APP_NAME=MyWebApp
APP_VERSION=1.0.0
LOG_FORMAT=json
```

**Environment-specific** (`.env.development`):
```bash
PORT=3000
DEBUG=true
DATABASE_URL=postgres://localhost:5432/myapp_dev
```

**Update `.ee`** to stack them:
```json
{
  "environments": {
    "development": {
      "sheets": [
        ".env",              // Base config
        ".env.development"   // Dev overrides
      ]
    },
    "production": {
      "sheets": [
        ".env",              // Base config
        ".env.production"    // Prod overrides
      ]
    }
  }
}
```

## Step 5: Export and Share

### Export to Different Formats

```bash
# Export as .env file
ee apply development --dry-run --format dotenv > .env.local

# Export as JSON
ee apply production --dry-run --format json > config.json

# Export as environment variables (for shell sourcing)
ee apply development --dry-run --format env > env.sh
```

### Use with Docker

```bash
# Export for Docker
ee apply production --dry-run --format dotenv > .env.docker

# Use in docker-compose
docker-compose --env-file .env.docker up
```

## Step 6: View Current Environment

Use the root `ee` command to inspect your current shell environment:

```bash
# Show all environment variables
ee

# Filter specific variables
ee --filter 'NODE*,PORT,DEBUG'

# Show as JSON
ee --format json

# Mask sensitive values
ee --filter '*KEY*,*SECRET*,*TOKEN*' --mask
```

## Working with Remote Schemas and Sheets

### Configure Remote URL

Edit your `.ee` file to add a remote URL:

```json
{
  "project": "my-web-app",
  "remote": "https://api.ee.dev",
  "schema": {
    "ref": "local://web-service"
  },
  "environments": { ... }
}
```

### Pull Shared Resources

```bash
# Pull schemas from remote
ee pull --schemas

# Pull config sheets from remote
ee pull --sheets

# Pull everything
ee pull
```

### Push Your Changes

```bash
# Push schemas to remote
ee push --schemas

# Push config sheets to remote
ee push --sheets

# Push everything
ee push
```

## Common Workflows

### Development Workflow

```bash
# 1. Initialize project
cd my-project
ee init

# 2. Edit development configuration
vim .env.development

# 3. Run application
ee apply development -- npm run dev

# 4. Run tests
ee apply development -- npm test
```

### Production Deployment

```bash
# 1. Preview production config
ee apply production --dry-run

# 2. Export for deployment
ee apply production --dry-run --format dotenv > .env.production

# 3. Deploy
ee apply production -- ./deploy.sh
```

### Team Collaboration

```bash
# 1. Pull latest schemas and sheets
ee pull

# 2. Work on your feature
ee apply development -- npm start

# 3. Share your schema changes
ee push --schemas
```

## Quick Reference

### Project Setup
```bash
ee init                              # Initialize project in current directory
ee init my-project                   # Initialize with specific name
ee init --schema web-service         # Initialize with existing schema
ee init --remote https://api.ee.dev  # Initialize with remote URL
```

### Schema Management
```bash
ee schema create <name>              # Create schema interactively
ee schema list                       # List all schemas
ee schema show <name>                # Show schema details
ee schema edit <name>                # Edit schema
ee schema delete <name>              # Delete schema
```

### Config Sheet Management
```bash
ee sheet create <name> --schema <schema>  # Create config sheet
ee sheet list                             # List all sheets
ee sheet show <name>                      # Show sheet details
ee sheet edit <name>                      # Edit sheet
ee sheet set <name> KEY value             # Set individual value
ee sheet export <name> --format env       # Export sheet
```

### Environment Management
```bash
ee apply <env>                       # Start shell with environment
ee apply <env> -- <command>          # Run command with environment
ee apply <env> --dry-run             # Preview environment
ee apply .env.file                   # Apply .env file directly
```

### Inspection
```bash
ee                                   # Show all current env vars
ee --filter 'PATTERN*'              # Filter env vars
ee --format json                     # Output as JSON
ee --mask                            # Mask sensitive values
```

### Remote Sync
```bash
ee pull                              # Pull all from remote
ee pull --schemas                    # Pull only schemas
ee pull --sheets                     # Pull only sheets
ee push                              # Push all to remote
ee push --schemas                    # Push only schemas
ee push --sheets                     # Push only sheets
```

## Tips

1. **Always use dry-run first**: `ee apply production --dry-run` before deploying
2. **Never commit secrets**: Keep sensitive `.env` files in `.gitignore`
3. **Use sheet stacking**: Combine base config with environment-specific overrides
4. **Leverage schemas**: Define validation rules to catch errors early
5. **Remote sync for teams**: Use push/pull for sharing schemas and config sheets
