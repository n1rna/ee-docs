Configuration sheets (or "sheets" for short) are where the actual environment variable values are stored. While schemas define the structure and validation rules, sheets contain the real data that gets loaded into your application's environment.

## What are Configuration Sheets?

A configuration sheet is a collection of key-value pairs that:
- Contains actual values for environment variables
- Is validated against a schema
- Can inherit values from other sheets
- Can exist standalone or be part of a project environment

Think of sheets as "instances" of a schema - they provide the concrete values that your application will use.

## Types of Configuration Sheets

### Standalone Sheets
Independent configuration sheets not tied to any project:

```bash
# Create a standalone sheet
ee sheet create database-config --schema database-schema

# Edit the sheet
ee sheet edit database-config

# Apply the sheet
ee apply database-config -- node server.js
```

### Project Environment Sheets
Sheets that belong to a specific project environment (covered in [Environments](/concepts/environments)):

```bash
# These are managed through project commands
ee project env edit my-app development
ee project env edit my-app production
```

## Sheet Structure

Here's what a configuration sheet looks like:

```yaml
# Sheet metadata
name: my-app-development
description: Development environment configuration for my web app
schema: web-service  # Reference to the schema
project: my-app      # Optional: associated project
environment: development  # Optional: environment name

# The actual variable values
values:
  PORT: "3000"
  DATABASE_URL: "postgres://user:pass@localhost:5432/myapp_dev"
  DEBUG: "true"
  API_KEY: "dev-api-key-123"
  LOG_LEVEL: "debug"

# Optional: inherit from other sheets
extends: ["base-config"]
```

## Managing Standalone Sheets

### Creating Sheets
```bash
# Create a new sheet with a schema
ee sheet create my-config --schema web-service

# Create a sheet with inline schema
ee sheet create custom-config --inline
```

### Listing Sheets
```bash
# List all standalone sheets
ee sheet list

# Search for specific sheets
ee sheet list --search database
```

### Editing Sheets
```bash
# Edit sheet values
ee sheet edit my-config

# Set individual values
ee sheet set my-config PORT=4000 DEBUG=false

# Unset values
ee sheet unset my-config DEBUG
```

### Viewing Sheet Contents
```bash
# Show sheet details
ee sheet show my-config

# Show resolved values (including inherited values)
ee sheet show my-config --resolved

# Show only the values
ee sheet export my-config --format env
```

### Applying Sheets
```bash
# Load into current shell
ee apply my-config

# Run a command with the sheet loaded
ee apply my-config -- npm start

# Export to different formats
ee export my-config --format json > config.json
ee export my-config --format yaml > config.yaml
ee export my-config --format env > .env
```

## Sheet Inheritance

Sheets can inherit values from other sheets using the `extends` field:

```yaml
# base-config sheet
name: base-config
schema: web-service
values:
  PORT: "3000"
  DEBUG: "false"
  LOG_LEVEL: "info"

---

# development-config sheet
name: development-config
schema: web-service
extends: ["base-config"]
values:
  DEBUG: "true"        # Overrides base-config
  LOG_LEVEL: "debug"   # Overrides base-config
  DATABASE_URL: "postgres://localhost:5432/dev"  # New value
  # PORT: "3000" is inherited from base-config
```

### Inheritance Rules
1. **Values are merged** from parent sheets to child sheets
2. **Child values override** parent values for the same variable
3. **Multiple inheritance** is supported with left-to-right precedence
4. **Circular references** are detected and prevented

### Multiple Inheritance Example
```yaml
name: production-config
extends: ["base-config", "security-config", "performance-config"]
values:
  # This sheet's values take precedence over all parent sheets
  NODE_ENV: "production"
```

## Schema Validation

All sheet values are validated against their associated schema:

### Validation Rules
- **Type checking**: String, number, boolean, URL validation
- **Required fields**: Must be present and non-empty
- **Regex patterns**: Custom validation rules
- **Default values**: Applied when variables are missing

### Validation Examples
```bash
# This will fail if PORT is not a valid number
ee sheet set my-config PORT=invalid

# This will fail if DATABASE_URL doesn't match the regex pattern
ee sheet set my-config DATABASE_URL=invalid-url

# This will fail if JWT_SECRET is required but not provided
ee sheet set my-config JWT_SECRET=""
```

## Dynamic Schemas

For advanced use cases, sheets can define their own inline schemas:

```yaml
name: custom-sheet
schema:
  # Inline schema definition
  variables:
    - name: CUSTOM_VAR
      type: string
      required: true
    - name: CUSTOM_PORT
      type: number
      default: "8080"
      required: false
values:
  CUSTOM_VAR: "my-value"
  CUSTOM_PORT: "9000"
```

This is useful for:
- One-off configurations
- Testing new variable structures
- Migrating between schema versions

## Working with Large Sheets

### Organizing Variables
Use consistent naming conventions:
```yaml
values:
  # Database variables
  DB_HOST: "localhost"
  DB_PORT: "5432"
  DB_NAME: "myapp"

  # Redis variables
  REDIS_HOST: "localhost"
  REDIS_PORT: "6379"

  # API variables
  API_KEY: "secret"
  API_BASE_URL: "https://api.example.com"
```

### Using Comments
Add YAML comments for documentation:
```yaml
values:
  # Database connection - uses local PostgreSQL in development
  DATABASE_URL: "postgres://user:pass@localhost:5432/myapp_dev"

  # Debug mode - enables verbose logging and error details
  DEBUG: "true"

  # Feature flags
  FEATURE_NEW_AUTH: "true"  # Enable new authentication system
  FEATURE_BETA_UI: "false"  # Beta UI is not ready for development
```

## Common Use Cases

### Database Configuration Sheet
```yaml
name: postgres-config
schema: database-schema
values:
  DB_HOST: "localhost"
  DB_PORT: "5432"
  DB_NAME: "myapp"
  DB_USER: "postgres"
  DB_PASSWORD: "secret123"
  DB_SSL: "false"
  DB_MAX_CONNECTIONS: "10"
```

### Development vs Production Sheets
```yaml
# development-sheet
name: myapp-dev
schema: web-service
values:
  PORT: "3000"
  DATABASE_URL: "postgres://localhost:5432/myapp_dev"
  DEBUG: "true"
  LOG_LEVEL: "debug"
  REDIS_URL: "redis://localhost:6379"

---

# production-sheet
name: myapp-prod
schema: web-service
values:
  PORT: "80"
  DATABASE_URL: "postgres://prod-db:5432/myapp"
  DEBUG: "false"
  LOG_LEVEL: "warn"
  REDIS_URL: "redis://prod-redis:6379"
```

### Shared Configuration Sheet
```yaml
# Base configuration shared across environments
name: shared-config
schema: base-schema
values:
  APP_NAME: "MyApp"
  APP_VERSION: "1.0.0"
  SUPPORTED_LANGUAGES: "en,es,fr"
  MAX_UPLOAD_SIZE: "10485760"  # 10MB
  SESSION_TIMEOUT: "3600"      # 1 hour
```

## Best Practices

### Security
- **Never commit sensitive values** to version control
- **Use different sheets** for different security levels
- **Rotate secrets regularly** and update sheets accordingly
- **Use minimal permissions** for sheet access

### Organization
- **Use descriptive names** that indicate purpose and environment
- **Group related variables** with consistent prefixes
- **Document complex values** with YAML comments
- **Keep sheets focused** - don't mix unrelated configurations

### Maintenance
- **Regular audits** - review and clean up unused variables
- **Schema alignment** - ensure sheets match their schemas
- **Inheritance chains** - keep inheritance simple and shallow
- **Value validation** - test configuration changes thoroughly
